    Redis未授权

    内网安全 2022 内网 渗透测试
    发布时间 : 2022-03-10 14:30
    阅读 :
    1. 写入Webshell文件
    2. 写入SSH公钥
    3. 写入计划任务
    4. 主从复制RCE

    Redis未授权

    ​ 1、写入ssh公钥（root运行redis）

    ​ 2、bind地址为0.0.0.0

    ​ 3、未开启密码保护

    ⭐利用条件：

    ​ 1、redis绑定在 0.0.0.0:6379，且没有进行添加防火墙规则避免其他非信任来源ip访问等相关安全策略，直接暴露在公网。

    ​ 2、没有设置密码认证（一般为空），可以免密码远程登录redis服务。

    ​ 3、redis使用root运行（service redis start是以redis普通账户运行的，redis-server /etc/redis/redis.conf #以root账户运行redis）

    Kali上用redis-cli连接

    ​ 无密码连接：redis-cli -h host -p port

    ​ 有密码连接：redis-cli -h hostip -p port -a password

    ​ 使用 ” ping “ 命令检查连接是否成功，回显 ” PONG “则连接成功

    写入Webshell文件

    ⭐利用条件：

    ​ 1、redis未授权，能用redis-cli连上

    ​ 2、开了web服务器，并且知道路径（如利用phpinfo，或者错误爆路经），还需要具有文件读写增删改查权限。

    ​ 3、我们可以将dir设置为一个目录a，而dbfilename为文件名b，再执行save或bgsave，则可以任意写入一个路径为a/b的文件。

    ​ 4、实列如下：写入/root/下失败（不是以root运行的不能访问），其它路径成功

    写入SSH公钥

    ⭐利用条件：

    ​ 服务端的Redis连接存在未授权，在攻击机上能用redis-cli直接登陆连接，并未登陆验证。

    ​ 服务端存在.ssh目录并且有写入的权限。

    ​ 必须是以root账户运行，redis-server /etc/redis/redis.conf

    ​ Redis主机上root目录下有“ ./ssh ”目录

    原理：数据库中插入一条数据，将本机的公钥作为value，key值随意，然后通过修改数据库的默认路径为/root/.ssh和默认的缓冲文件authorized.keys，把缓冲的数据保存在文件里，这样就可以在服务器端的/root/.ssh下生成一个授权的key。

    注意：将所有缓冲区的内容下入到authorized.keys文件，只要公钥的内容没被破坏（也不可能破坏）就可以连接上

    1、自己电脑生成公钥文件（Kali上执行）

    2、将公钥写入Redis中

    ​ 说明：将公钥导入key.txt文件(前后用\n换行，避免和Redis里其他缓存数据混合)，再把key.txt文件内容写入目标主机的缓冲里：

    ​ (echo -e "\n\n"; cat 公钥文件; echo -e "\n\n") > key.txt

    3、将数据保存到目标redis硬盘上

    ​ 说明：使用攻击机连接目标机器Redis，设置Redis的备份路径为/root/.ssh/并保存文件名为authorized_keys，并将数据保存在目标服务器硬盘上

    写入计划任务

    ⭐利用条件：

    ​ 服务端的Redis连接存在未授权，在攻击机上能用redis-cli直接登陆连接，并未登陆验证

    ​ Root权限运行Redis服务（可选）

    原理：

    ​ 1、在数据库中插入一条数据，将计划任务的内容作为value，key值随意，

    ​ 2、然后通过修改数据库的默认路径为目标主机计划任务的路径，把缓冲的数据保存在文件里，可以在服务器端成功写入一个计划任务进行反弹shell。

    使用限制：

    ​ Centos的定时任务文件在/var/spool/cron/

    ​ Ubuntu定时任务文件在/var/spool/cron/crontabs/﻿

    【01-主从复制RCE】

    PS：未安装成功，未作记录

    背景：
    利用 crontab、ssh key、webshell 这三种方式具有一定的容错性，同时 crontab、ssh 服务是服务器的标准命令和服务，因此很多情况下这种通过写文件的方式 getshell 是通杀的。但随着现代服务部署方式的不断发展，组件化成了不可逃避的大趋势。docker就是在该时代背景下的产物之一，在这种部署模式下，一个单一的容器中不会有除了 redis 以外的任何服务存在，包括 ssh 和 crontab，再加上权限的严格控制，所以单纯依靠写文件就很难getshell，在这种情况下我们就需要其他的利用手段。
    产生：
    Redis 在 4.x、5.x 版本中提供了主从模式，主从模式指的是使用单个 redis 为主机，而将其他 redis 作为从机，主机和从机的数据都是一样的，主机只负责写，从机只负责读。在 Redis 4.x 版本后，通过外部扩展可以实现一个新的 Redis 命令来构造恶意 .so 文件，在两个 Redis 实例中设置主从模式的时候，主机可以通过 FULLRESYNC 同步文件到从机上，然后在从机上加载恶意 .so 文件即可执行命令。简单的说就是攻击者（主机）写一个 .so 文件，通过 FULLRESYNC 同步文件到受害者（从机）上。
    ⭐利用条件：
    Redis版本：4.x，< 5.0.5
